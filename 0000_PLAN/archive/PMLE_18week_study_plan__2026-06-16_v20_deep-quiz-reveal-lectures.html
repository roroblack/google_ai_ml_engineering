복습: Getting Started with BigQuery ML

  • 15m 📝 핵심 개념 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답·재현
📌 핵심 포인트
  • CREATE MODEL 문은 SQL만으로 BigQuery 내부에서 ML 모델을 학습시키며, 데이터를 외부로 이동할 필요가 없다. OPTIONS 절에서 model_type, input_label_cols 등 하이퍼파라미터를 지정한다.
  • model_type='logistic_reg'는 분류(이진/다중) 문제에, model_type='linear_reg'는 연속형 수치를 예측하는 회귀 문제에 사용한다. label 컬럼의 자료형이 아니라 풀려는 문제의 성격으로 선택한다.
  • input_label_cols로 label(정답) 컬럼을 명시하지 않으면 BQML은 기본적으로 'label'이라는 이름의 컬럼을 찾는다. SELECT 결과의 다른 모든 컬럼은 feature로 사용된다.
  • ML.EVALUATE는 모델 성능 지표를 반환하고, ML.PREDICT는 새 데이터에 대한 예측값을 반환한다. ML.PREDICT의 출력 컬럼명은 predicted_<label> 형식이다.
  • train/eval 분할은 과적합 평가를 위해 필수이며, BQML은 OPTIONS의 data_split_method(예: AUTO_SPLIT, RANDOM)로 자동 분할하거나 WHERE 절로 수동 분할할 수 있다.
  • 학습 데이터에 NULL이 있는 행은 자동으로 제외되며, 학습 시 feature는 자료형에 따라 자동으로 전처리(예: 문자열은 one-hot 인코딩)된다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. 연속형 값인 주택 가격을 예측하는 모델을 BQML로 만들 때 OPTIONS에 지정해야 할 model_type은?
    1. A. logistic_reg
    2. B. linear_reg
    3. C. kmeans
    4. D. boosted_tree_classifier
    정답 B. 연속형 수치를 예측하는 것은 회귀(regression) 문제이므로 linear_reg가 정답이다. logistic_reg는 범주(클래스)를 예측하는 분류용이며 가격 같은 연속값에는 부적합하고, kmeans는 비지도 군집화 알고리즘이다.
  2. CREATE MODEL 문에서 input_label_cols를 지정하지 않았을 때 BQML이 label로 간주하는 컬럼은?
    1. A. SELECT의 첫 번째 컬럼
    2. B. 이름이 'label'인 컬럼
    3. C. 이름이 'target'인 컬럼
    4. D. SELECT의 마지막 컬럼
    정답 B. input_label_cols를 생략하면 BQML은 기본적으로 'label'이라는 이름의 컬럼을 정답으로 사용한다. 첫 번째나 마지막 컬럼을 위치로 추정하지 않으므로, 컬럼명이 다르면 input_label_cols로 명시해야 한다.
  3. 학습된 분류 모델의 정밀도, 재현율 등 성능 지표를 확인하려면 어떤 함수를 사용하는가?
    1. A. ML.PREDICT
    2. B. ML.EVALUATE
    3. C. ML.TRAINING_INFO
    4. D. ML.FEATURE_INFO
    정답 B. ML.EVALUATE는 평가 데이터에 대해 모델의 성능 지표(precision, recall, accuracy 등)를 반환한다. ML.PREDICT는 개별 행의 예측값을 반환할 뿐 집계 지표를 주지 않으며, ML.TRAINING_INFO는 반복별 손실 등 학습 과정 정보를 제공한다.
  4. BQML에서 학습/평가 데이터를 나누지 않고 전체 데이터로 학습한 뒤 같은 데이터로 평가하면 생기는 문제는?
    1. A. 모델 학습이 실패한다
    2. B. 과적합 여부를 알 수 없어 일반화 성능이 과대평가된다
    3. C. label 컬럼이 자동 삭제된다
    4. D. 예측 속도가 느려진다
    정답 B. 학습에 사용한 데이터로 평가하면 모델이 외운 패턴 때문에 지표가 비현실적으로 좋게 나와 새 데이터에 대한 일반화 성능을 알 수 없다. 학습이 실패하거나 컬럼이 삭제되는 일은 없으며, 이는 정확도가 아니라 신뢰성의 문제다.
  5. ML.PREDICT를 logistic_reg 모델에 사용했을 때 출력에 포함되는 예측 결과 컬럼 형식으로 옳은 것은?
    1. A. result라는 단일 컬럼만 출력
    2. B. predicted_<label>과 클래스별 확률 컬럼
    3. C. 원본 label 컬럼을 그대로 덮어씀
    4. D. error_rate 컬럼
    정답 B. ML.PREDICT는 predicted_<label> 컬럼과 함께 각 클래스에 대한 확률(predicted_<label>_probs)을 반환한다. 원본 label을 덮어쓰지 않고 입력 컬럼과 예측 컬럼을 함께 출력하며, error_rate 같은 컬럼은 제공하지 않는다.

복습: Predict Visitor Purchases with a Classification Model in BigQuery ML

  • 15m 📝 핵심 개념 정리
  • 25m ❓ 퀴즈 5문항 풀이

📌 핵심 포인트
  • 이진 분류는 logistic_reg로 학습하며, 방문자가 구매(1)할지 안 할지(0)를 예측한다. GA(Google Analytics) e-commerce 데이터처럼 대부분의 세션이 구매로 이어지지 않는 클래스 불균형 상황이 흔하다.
  • ML.EVALUATE는 precision, recall, accuracy, f1_score, log_loss, roc_auc를 반환한다. 불균형 데이터에서는 accuracy가 다수 클래스에 치우쳐 오해를 주므로 roc_auc나 precision/recall, f1을 함께 봐야 한다.
  • precision = TP/(TP+FP)는 양성 예측 중 맞은 비율, recall = TP/(TP+FN)는 실제 양성 중 잡아낸 비율이다. 둘은 트레이드오프 관계이며 f1_score는 둘의 조화평균이다.
  • 분류 threshold(기본 0.5)를 조정하면 precision과 recall의 균형을 바꿀 수 있다. threshold를 높이면 precision↑·recall↓, 낮추면 recall↑·precision↓ 경향이 있다.
  • roc_auc는 threshold와 무관하게 모델이 양성과 음성을 얼마나 잘 구분하는지를 나타내며, 1.0에 가까울수록 좋고 0.5는 무작위 추측 수준이다.
  • log_loss는 예측 확률의 정확도를 벌점화하는 지표로 낮을수록 좋으며, 확신을 가지고 틀린 예측에 큰 페널티를 준다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. 구매 전환이 전체 세션의 약 2%인 매우 불균형한 데이터에서 모델 성능을 평가할 때 가장 신뢰하기 어려운 단일 지표는?
    1. A. roc_auc
    2. B. accuracy
    3. C. recall
    4. D. f1_score
    정답 B. 불균형 데이터에서는 모든 세션을 '구매 안 함'으로 예측해도 accuracy가 98%에 달해 모델이 좋아 보이는 착시를 준다. roc_auc, recall, f1은 소수 양성 클래스의 탐지 능력을 반영하므로 이런 상황에서 더 적절하다.
  2. 실제 구매자를 최대한 놓치지 않는 것(누락 최소화)이 사업적으로 가장 중요할 때 우선 최적화해야 할 지표는?
    1. A. precision
    2. B. recall
    3. C. log_loss
    4. D. specificity
    정답 B. recall = TP/(TP+FN)은 실제 양성 중 모델이 잡아낸 비율로, 놓침(False Negative)을 줄이는 데 직결된다. precision은 양성 예측의 정확성에 관한 것이라 누락 최소화 목표와는 초점이 다르다.
  3. 분류 threshold를 0.5에서 0.8로 높이면 일반적으로 나타나는 변화는?
    1. A. precision 증가, recall 감소
    2. B. precision 감소, recall 증가
    3. C. 둘 다 증가
    4. D. roc_auc가 감소
    정답 A. threshold를 높이면 더 확신할 때만 양성으로 분류하므로 양성 예측의 정확성(precision)은 올라가지만, 애매한 실제 양성을 놓쳐 recall은 내려간다. roc_auc는 threshold와 무관한 지표이므로 threshold 변경으로 바뀌지 않는다.
  4. roc_auc 값이 0.5 부근으로 나왔다면 이는 무엇을 의미하는가?
    1. A. 모델이 완벽하게 분류한다
    2. B. 모델이 무작위 추측 수준으로 양성/음성을 구분하지 못한다
    3. C. 데이터에 NULL이 많다
    4. D. threshold가 잘못 설정됐다
    정답 B. roc_auc는 1.0이 완벽한 구분, 0.5가 동전 던지기 수준의 무작위 성능을 의미한다. 따라서 0.5 부근은 모델이 클래스를 구별하는 능력이 거의 없다는 뜻이며, NULL이나 threshold 설정과는 직접 관련이 없다.
  5. 두 모델이 비슷한 accuracy를 보이지만 한 모델의 log_loss가 훨씬 낮다면 이를 어떻게 해석해야 하는가?
    1. A. log_loss가 낮은 모델이 예측 확률의 신뢰도(calibration)가 더 좋다
    2. B. 두 모델은 완전히 동일하다
    3. C. log_loss가 낮은 모델은 과적합되었다
    4. D. accuracy가 같으면 log_loss는 무시해도 된다
    정답 A. log_loss는 예측 확률 자체의 품질을 평가하며, 낮을수록 확률 예측이 실제 결과에 잘 맞춰져 있음을 뜻한다. accuracy는 같아도 확신을 가지고 틀린 예측이 많으면 log_loss가 커지므로, 낮은 쪽이 확률적으로 더 신뢰할 만하다.

  • 택시 요금 같은 연속형 값 예측은 회귀 문제이며 model_type='linear_reg'로 학습한다. ML.EVALUATE는 회귀 지표로 RMSE, MAE, MSE, R² 등을 반환한다.
  • RMSE(평균제곱근오차)는 오차를 제곱하므로 큰 오차(이상치)에 민감하고, MAE(평균절대오차)는 모든 오차를 동등하게 취급해 이상치에 더 강건하다.
  • 비현실적인 데이터(예: 음수 요금, 0 거리, 위경도 0,0 같은 결측 좌표, 비정상적으로 먼 거리)는 WHERE 절로 필터링해 outlier를 제거해야 모델 품질이 향상된다.
  • feature selection은 예측에 의미 있는 컬럼만 사용하는 것이 핵심이며, 누수(leakage)를 일으키는 컬럼(예: 결제 후에야 알 수 있는 팁·총액)을 feature로 넣으면 안 된다.
  • feature cross(ML.FEATURE_CROSS)는 승차/하차 위치, 시간대 등 개별 feature를 조합해 비선형 상호작용을 선형 모델이 학습하도록 돕는다. 위경도를 버킷화한 뒤 교차하면 지역 효과를 포착한다.
  • 학습 전후에 train/eval을 분리하고 동일 전처리를 적용해야 하며, RMSE를 평가셋에서 비교해 feature engineering의 효과를 확인한다.

  • TRANSFORM 절은 CREATE MODEL 안에서 feature 전처리를 정의하며, 그 전처리 로직이 모델에 함께 저장된다. 따라서 ML.PREDICT 시 동일 전처리가 자동 적용되어 training-serving skew를 방지한다.
  • ML.BUCKETIZE는 연속형 값을 지정한 경계(분위/구간)에 따라 범주형 버킷으로 나눠 비선형 관계를 선형 모델이 다루게 한다.
  • ML.STANDARD_SCALER는 수치 feature를 평균 0, 표준편차 1로 표준화해 스케일이 다른 feature들이 공정하게 학습되도록 한다.
  • ML.FEATURE_CROSS는 둘 이상의 범주형(또는 버킷화된) feature를 조합해 상호작용 항을 만들어 선형 모델이 feature 간 결합 효과를 학습하게 한다.
  • TRANSFORM을 사용하면 예측 시 raw 입력만 넣으면 되고 별도의 전처리 쿼리를 재작성할 필요가 없어, 학습과 서빙 간 전처리 불일치 위험이 사라진다.
  • training-serving skew는 학습 때와 예측 때 데이터 처리 방식이 달라 성능이 저하되는 현상이며, TRANSFORM 절이 이를 막는 BQML의 핵심 메커니즘이다.

복습: Performing Basic Feature Engineering in Keras

  • 15m 📝 핵심 개념 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답·재현
📌 핵심 포인트
  • Keras에서는 feature_column API 또는 최신 preprocessing layer(예: Normalization, StringLookup, Discretization)로 입력 feature를 모델 그래프에 통합해 전처리한다.
  • 수치형 feature는 Normalization 레이어 등으로 정규화하며, 평균과 분산은 학습 데이터에서 adapt로 계산해 학습-서빙 일관성을 유지한다.
  • 범주형 feature 인코딩에서 카디널리티가 낮으면 one-hot(indicator)이 적합하고, 카디널리티가 매우 높으면 메모리·일반화 측면에서 embedding(밀집 벡터)이 적합하다.
  • embedding은 범주를 저차원 밀집 벡터로 학습해 유사한 범주가 가까운 표현을 갖게 하며, one-hot의 희소·고차원 문제를 완화한다.
  • feature cross는 crossed_column 또는 HashedCrossing 등으로 여러 범주형 feature를 조합해 상호작용을 모델이 학습하도록 하며, 고카디널리티 교차는 해시 버킷으로 차원을 제한한다.
  • 전처리를 모델 입력 파이프라인(또는 레이어)에 포함하면 raw 입력으로 학습·서빙이 가능해 BQML의 TRANSFORM처럼 skew를 줄인다.

    2. B. 유사한 범주가 가까운 벡터를 갖도록 학습되어 의미적 관계를 표현한다
    3. C. 전처리가 전혀 필요 없다
    4. D. 범주 수에 비례해 차원이 늘어난다
    정답 B. embedding은 범주를 밀집 벡터 공간에 학습 배치해 유사한 범주끼리 가까운 표현을 갖게 하고, 고정된 저차원으로 고카디널리티를 다룬다. one-hot은 범주 수에 비례해 차원이 커지는 반면 embedding은 그렇지 않다.

    3. C. label을 늘린다

복습: Classifying Structured Data using Keras Preprocessing Layers

  • 15m 📝 전처리 layer·adapt 핵심 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답 복습 및 파이프라인 재현
📌 핵심 포인트
  • Keras preprocessing layer는 모델 그래프 내부에 전처리를 포함시켜 training과 serving에서 동일한 변환을 보장하므로 training-serving skew를 구조적으로 방지한다.
  • Normalization은 수치형 feature를 평균 0, 분산 1로 표준화하고, adapt()를 통해 학습 데이터에서 mean/variance 같은 상태(state)를 미리 계산해 둔다.
  • StringLookup은 문자열 범주를 정수 인덱스로, IntegerLookup은 정수 범주를 인덱스로 매핑하며, OOV(out-of-vocabulary) 토큰과 vocabulary를 adapt()로 생성한다.
  • CategoryEncoding은 정수 인덱스를 one-hot/multi-hot/count 형태로 인코딩하므로 보통 StringLookup·IntegerLookup 뒤에 연결해 사용한다.
  • tf.data.Dataset으로 (features dict, label) 형태를 구성하고 batch/prefetch로 입력 파이프라인을 효율화하며, pandas DataFrame을 from_tensor_slices로 변환한다.
  • functional API는 각 입력 feature마다 keras.Input을 정의하고 전처리 layer를 연결한 뒤 하나의 모델로 합치므로, 이질적인 구조적 데이터를 다루기에 적합하다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. Normalization layer를 사용할 때 adapt() 단계가 반드시 필요한 이유는 무엇인가?
    1. A. 모델 가중치를 무작위로 초기화하기 위해
    2. B. 학습 데이터로부터 mean과 variance 같은 정규화 상태를 미리 계산하기 위해
    3. C. label을 one-hot으로 변환하기 위해
    4. D. optimizer의 learning rate를 자동 조정하기 위해
    정답 B. Normalization은 입력을 표준화하기 위해 mean·variance라는 상태가 필요하며 adapt()가 데이터를 한 번 훑어 이를 계산한다. 가중치 초기화나 learning rate 조정은 전처리 layer의 adapt()와 무관하므로 오답이다.
  2. 전처리 로직을 별도 스크립트가 아니라 Keras preprocessing layer로 모델 안에 포함시키는 가장 큰 이점은?
    1. A. 학습 속도가 항상 GPU 없이도 빨라진다
    2. B. training과 serving에서 동일한 변환이 적용되어 training-serving skew를 방지한다
    3. C. 모델 파라미터 수가 줄어 메모리를 절약한다
    4. D. label이 없어도 비지도 학습이 가능해진다
    정답 B. 전처리가 모델 그래프의 일부가 되면 배포 시에도 같은 변환이 그대로 수행되어 학습-서빙 간 불일치(skew)를 막는다. 속도·파라미터·비지도 학습은 이 설계의 핵심 목적이 아니다.
  3. 문자열 범주형 feature(예: 'cat','dog')를 모델 입력으로 처리하려 한다. 올바른 layer 연결 순서는?
    1. A. Normalization 다음 CategoryEncoding
    2. B. StringLookup 다음 CategoryEncoding
    3. C. IntegerLookup 다음 Normalization
    4. D. CategoryEncoding 다음 StringLookup
    정답 B. 문자열은 먼저 StringLookup으로 정수 인덱스로 매핑한 뒤 CategoryEncoding으로 one-hot/multi-hot 인코딩한다. Normalization은 수치형용이고 순서를 뒤집으면 인코딩 대상이 없어 동작하지 않는다.
  4. pandas DataFrame을 Keras 입력 파이프라인에 적합한 형태로 만들 때 적절한 방법은?
    1. A. DataFrame을 그대로 model.fit에 넣으면 batch/prefetch가 자동 적용된다
    2. B. tf.data.Dataset.from_tensor_slices로 (features dict, label) 형태를 만들고 batch·prefetch를 적용한다
    3. C. feature를 모두 문자열로 합쳐 단일 컬럼으로 만든다
    4. D. label을 feature에 포함시켜 하나의 텐서로 전달한다
    정답 B. from_tensor_slices로 feature dict와 label을 묶은 Dataset을 만든 후 batch와 prefetch로 입력 효율을 높이는 것이 표준 패턴이다. label을 feature에 섞으면 supervised 학습이 불가능하므로 오답이다.
  5. 학습 데이터에 없던 새로운 범주 값이 serving 시 들어올 때 StringLookup이 이를 처리하는 방식은?
    1. A. 에러를 발생시켜 예측을 중단한다
    2. B. OOV(out-of-vocabulary) 토큰 인덱스로 매핑한다
    3. C. 가장 빈도 높은 범주로 자동 대체한다
    4. D. 해당 행을 자동으로 삭제한다
    정답 B. StringLookup은 vocabulary에 없는 값을 OOV 토큰 인덱스로 보내 안전하게 처리한다. 에러 중단이나 행 삭제는 기본 동작이 아니며 빈도 기반 대체도 수행하지 않는다.

복습: Introducing the Keras Sequential API on Agent Platform

  • 15m 📝 API 종류·loss/activation 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답 복습 및 compile/fit 재현
📌 핵심 포인트
  • Sequential API는 layer를 일렬로 쌓는 단순 모델에 적합하고, 다중 입력·출력이나 분기·공유 layer가 필요하면 functional API를 사용한다.
  • Dense layer의 units는 출력 뉴런 수를, activation은 비선형성을 결정하며, 은닉층은 보통 relu, 이진 분류 출력은 sigmoid, 다중 분류 출력은 softmax를 쓴다.
  • compile()은 optimizer(예: adam), loss, metrics를 지정하여 학습 방식을 설정하는 단계다.
  • task별 loss 선택: 이진 분류는 binary_crossentropy, 다중 분류(정수 label)는 sparse_categorical_crossentropy, 회귀는 mse(MSE)를 사용한다.
  • fit()은 학습, evaluate()는 성능 평가, predict()는 추론을 수행하며 fit에는 epochs·batch_size·validation_data를 전달한다.
  • Agent Platform의 관리형 노트북·워크벤치 환경에서 모델을 개발·학습하고 이후 학습/서빙 서비스로 확장할 수 있다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. 두 개의 서로 다른 입력 소스를 받아 중간에서 합치는(branch & merge) 모델을 만들어야 한다. 적절한 선택은?
    1. A. Sequential API로 충분하다
    2. B. functional API를 사용해야 한다
    3. C. compile() 옵션만 바꾸면 된다
    4. D. predict()를 두 번 호출하면 된다
    정답 B. Sequential은 단일 입력·단일 출력의 선형 스택만 표현하므로 다중 입력·분기·병합은 functional API가 필요하다. compile이나 predict는 구조 표현과 무관하다.
  2. 10개 클래스를 정수 label로 분류하는 모델의 출력층과 loss로 올바른 조합은?
    1. A. units=1, sigmoid, binary_crossentropy
    2. B. units=10, softmax, sparse_categorical_crossentropy
    3. C. units=10, relu, mse
    4. D. units=1, softmax, categorical_crossentropy
    정답 B. 다중 클래스 분류는 클래스 수만큼의 출력에 softmax를 쓰고, 정수 label이면 sparse_categorical_crossentropy가 맞다. sigmoid·binary_crossentropy는 이진용이고 회귀용 mse는 분류에 부적합하다.
  3. compile() 단계에서 지정하지 않는 것은?
    1. A. optimizer
    2. B. loss function
    3. C. 평가 metrics
    4. D. 학습 epoch 수
    정답 D. compile은 optimizer, loss, metrics를 설정하는 단계이며 epoch 수는 fit() 호출 시 전달한다. 따라서 epoch는 compile의 인자가 아니다.
  4. 주택 가격처럼 연속적인 실수 값을 예측하는 회귀 모델에 적합한 출력 활성화와 loss는?
    1. A. softmax 출력, categorical_crossentropy
    2. B. 활성화 없음(linear) 출력, mse
    3. C. sigmoid 출력, binary_crossentropy
    4. D. relu 출력, sparse_categorical_crossentropy
    정답 B. 회귀는 임의의 실수를 출력해야 하므로 마지막 Dense에 활성화를 두지 않고(linear) MSE 같은 회귀 loss를 쓴다. softmax/sigmoid와 crossentropy 계열은 분류 전용이라 부적합하다.
  5. 학습이 끝난 모델의 일반화 성능을 별도 테스트셋으로 측정하려 할 때 사용하는 메서드는?
    1. A. fit()
    2. B. compile()
    3. C. evaluate()
    4. D. adapt()
    정답 C. evaluate()는 주어진 데이터에 대해 loss·metrics를 계산해 성능을 평가한다. fit은 학습, compile은 설정, adapt는 전처리 layer 상태 계산이므로 평가 용도가 아니다.

복습: Training at Scale with Agent Platform Training Service

  • 15m 📝 패키징·container·AIP_MODEL_DIR 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답 복습 및 task.py 인자 재현
📌 핵심 포인트
  • custom training job은 사용자가 작성한 학습 코드를 관리형 인프라에서 실행하여 로컬 머신 한계를 넘어 확장하고 자원을 자동 프로비저닝한다.
  • 코드 패키징은 보통 model.py(모델·학습 로직)와 task.py(엔트리포인트·argument 파싱)로 구성하고 Python source package 또는 container로 제출한다.
  • prebuilt container는 TensorFlow 등 공통 프레임워크에 빠르게 쓰고, custom container는 특수 의존성·완전한 환경 제어가 필요할 때 사용한다.
  • 학습 산출물(모델, 체크포인트)은 GCS 버킷에 저장하여 영속화하고 이후 서빙·평가에 재사용한다.
  • AIP_MODEL_DIR 환경변수는 학습 서비스가 모델 저장 위치로 주입하는 GCS 경로이므로 코드에서 이를 읽어 모델을 export해야 한다.
  • 하이퍼파라미터(learning rate, epoch 등)는 task.py에서 command-line argument로 받아 외부에서 주입 가능하게 하여 재현성과 튜닝을 지원한다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. 학습 코드에서 모델을 저장할 GCS 경로를 하드코딩하지 않고 학습 서비스가 주입하는 위치에 저장하려면 무엇을 읽어야 하는가?
    1. A. AIP_MODEL_DIR 환경변수
    2. B. PATH 환경변수
    3. C. model.py의 전역 상수
    4. D. 로컬 /tmp 디렉터리
    정답 A. 학습 서비스는 AIP_MODEL_DIR 환경변수로 모델 export 경로(GCS)를 주입하므로 코드가 이를 읽어 저장해야 산출물이 올바르게 영속화된다. /tmp나 전역 상수는 작업 종료 시 사라지거나 이식성이 떨어진다.
  2. task.py와 model.py로 코드를 분리하는 일반적인 이유는?
    1. A. task.py는 모델 구조를, model.py는 명령줄 인자를 담당하게 하려고
    2. B. task.py는 엔트리포인트·argument 파싱을, model.py는 모델·학습 로직을 담당하게 하려고
    3. C. 두 파일이 있어야만 GPU를 쓸 수 있어서
    4. D. GCS 접근에 두 파일이 필수라서
    정답 B. 관례상 task.py가 진입점에서 하이퍼파라미터 등 argument를 파싱하고, model.py가 모델 정의와 학습 로직을 담아 책임을 분리한다. 역할이 뒤바뀐 설명이나 GPU·GCS 필수 조건은 사실이 아니다.
  3. 표준 TensorFlow만 사용하는 학습 작업을 가장 빠르게 시작하려면?
    1. A. 직접 Dockerfile을 작성해 custom container를 빌드한다
    2. B. prebuilt training container를 사용한다
    3. C. 로컬 머신에서만 학습한다
    4. D. 모든 의존성을 GCS에 업로드한다
    정답 B. 공통 프레임워크는 prebuilt container로 환경 구성 없이 즉시 학습을 시작할 수 있다. custom container는 특수 의존성이 있을 때 유용하지만 표준 환경에서는 불필요한 추가 작업이다.
  4. 하이퍼파라미터를 task.py의 command-line argument로 받도록 설계하는 주된 이점은?
    1. A. 학습 속도가 자동으로 2배 빨라진다
    2. B. 코드 변경 없이 외부에서 값을 주입해 튜닝·재현이 쉬워진다
    3. C. GCS 버킷이 자동 생성된다
    4. D. loss function이 자동 선택된다
    정답 B. argument 주입은 코드를 수정하지 않고도 여러 설정을 실행할 수 있게 해 hyperparameter tuning과 실험 재현성을 높인다. 속도 향상이나 버킷·loss 자동화와는 관련이 없다.
  5. custom container를 prebuilt container 대신 선택해야 하는 상황은?
    1. A. 표준 TensorFlow 모델만 학습할 때
    2. B. PyPI에 없는 특수 라이브러리나 시스템 의존성이 필요해 환경을 완전히 제어해야 할 때
    3. C. 학습 데이터 크기가 작을 때
    4. D. 단순히 epoch 수를 늘리고 싶을 때
    정답 B. custom container는 비표준 의존성이나 정밀한 환경 제어가 필요할 때 적합하다. 표준 프레임워크·소규모 데이터·단순 설정 변경은 prebuilt container로 충분하다.

복습: Agent Platform: Training and Serving a Custom Model

  • 15m 📝 Registry·Endpoint·예측 방식 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답 복습 및 traffic split 재현
📌 핵심 포인트
  • Model Registry는 학습된 모델 버전을 등록·관리하는 중앙 저장소로, 배포 전 모델의 버전 관리와 거버넌스를 제공한다.
  • online prediction은 Endpoint에 모델을 배포해 낮은 지연으로 실시간 요청에 응답하고, batch prediction은 대량 데이터를 비동기로 처리하며 Endpoint가 필요 없다.
  • Endpoint 생성 후 모델을 deploy하면 추론을 위한 compute 자원이 프로비저닝되며, 하나의 Endpoint에 여러 모델을 배포할 수 있다.
  • traffic split은 한 Endpoint의 여러 배포 모델 간 트래픽 비율을 나눠 A/B 테스트나 점진적 롤아웃(canary)을 가능하게 한다.
  • prebuilt serving container는 표준 프레임워크 모델을 빠르게 서빙하고, custom serving container는 특수 전처리·런타임이나 비표준 입출력 형식이 필요할 때 사용한다.
  • 지연이 중요하지 않고 정기적으로 큰 데이터셋을 점수화할 때는 비용 효율적인 batch prediction이, 즉각 응답이 필요하면 online이 적합하다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. 수백만 건의 데이터를 야간에 한 번 점수화하며 실시간 응답은 필요 없다. 가장 적합한 예측 방식은?
    1. A. Endpoint를 통한 online prediction
    2. B. batch prediction
    3. C. Model Registry 직접 조회
    4. D. traffic split 설정
    정답 B. 대량 데이터를 비동기로 처리하고 지연이 중요치 않으면 Endpoint 없이 동작하는 batch prediction이 비용 효율적이다. online은 항상 켜진 Endpoint 자원이 필요해 이 시나리오엔 낭비다.
  2. 새 모델 버전을 전체 트래픽에 한 번에 노출하지 않고 10%에만 보내 점진적으로 검증하려 한다. 사용하는 기능은?
    1. A. Model Registry 버전 삭제
    2. B. 한 Endpoint에 두 모델을 배포하고 traffic split을 90/10으로 설정
    3. C. batch prediction job 두 개 실행
    4. D. AIP_MODEL_DIR 변경
    정답 B. 하나의 Endpoint에 기존·신규 모델을 함께 배포하고 traffic split으로 비율을 나누면 canary 배포·A/B 테스트가 가능하다. batch job이나 AIP_MODEL_DIR은 트래픽 비율 제어와 무관하다.
  3. online prediction을 제공하기 위해 모델 배포 시 반드시 필요한 것은?
    1. A. Endpoint 생성 및 모델 deploy
    2. B. batch prediction job 생성
    3. C. GCS 버킷 공개 설정
    4. D. 별도의 학습 작업 재실행
    정답 A. online 예측은 Endpoint를 만들고 그곳에 모델을 deploy하여 compute 자원을 띄워야 실시간 요청을 처리할 수 있다. batch job이나 학습 재실행은 online 서빙의 전제 조건이 아니다.
  4. Model Registry의 주된 역할로 가장 적절한 것은?
    1. A. 모델 학습 자체를 수행한다
    2. B. 학습된 모델 버전을 등록·관리하여 버전 관리와 배포 거버넌스를 제공한다
    3. C. 실시간 추론 트래픽을 직접 받는다
    4. D. 데이터 전처리를 자동 수행한다
    정답 B. Model Registry는 모델 버전의 중앙 카탈로그로 등록·버전 관리·배포 관리를 담당한다. 학습 실행·추론 트래픽 처리·전처리는 각각 학습 서비스·Endpoint·전처리 단계의 역할이다.
  5. 비표준 입력 형식을 받아 사용자 정의 전처리 후 추론해야 하는 모델을 서빙하려 한다. 적절한 선택은?
    1. A. prebuilt serving container
    2. B. custom serving container
    3. C. batch prediction 전용 모드
    4. D. Model Registry만으로 서빙
    정답 B. 표준 형식·프레임워크를 벗어난 전처리·런타임이 필요하면 custom serving container로 입출력과 환경을 직접 제어한다. prebuilt는 표준 케이스용이고 Registry는 서빙 자체를 수행하지 않는다.

복습: Distributed Training with Keras

  • 15m 📝 strategy 종류·all-reduce·TF_CONFIG 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답 복습 및 scope/batch 재현
📌 핵심 포인트
  • tf.distribute.Strategy는 학습 코드를 거의 바꾸지 않고 여러 device/머신에 분산 학습을 적용하는 API로, 대부분 data parallelism 방식을 쓴다.
  • MirroredStrategy는 단일 머신의 여러 GPU에 모델을 복제해 동기 학습하고, MultiWorkerMirroredStrategy는 여러 워커 머신으로 확장한다.
  • 모델 생성과 compile은 strategy.scope() 내부에서 수행해야 변수가 각 replica에 올바르게 복제·동기화된다.
  • data parallelism에서는 global batch size가 각 replica의 batch로 나뉘므로, replica 수를 늘리면 global batch size를 키우고 그에 맞춰 learning rate를 조정하는 것이 일반적이다.
  • all-reduce는 각 replica가 계산한 gradient를 모아 합산·평균해 모든 replica가 동일한 업데이트를 받도록 동기화하는 연산이다.
  • MultiWorkerMirroredStrategy는 TF_CONFIG 환경변수로 클러스터 구성(워커 목록·역할)을 인식하며, 각 워커가 동일한 학습 스크립트를 실행한다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. 단일 머신에 장착된 여러 GPU로 동기 분산 학습을 하려 한다. 가장 적합한 strategy는?
    1. A. MultiWorkerMirroredStrategy
    2. B. MirroredStrategy
    3. C. ParameterServerStrategy
    4. D. 분산 전략 없이 그냥 학습
    정답 B. MirroredStrategy는 단일 머신의 여러 GPU에 모델을 복제해 동기 data parallelism을 수행하는 표준 선택이다. MultiWorker는 여러 머신용으로 단일 노드에는 불필요하게 복잡하다.
  2. 모델 생성과 compile을 strategy.scope() 안에서 해야 하는 이유는?
    1. A. loss가 자동으로 선택되게 하려고
    2. B. 변수(weight)가 각 replica에 올바르게 복제·동기화되도록 하려고
    3. C. GCS 버킷이 자동 마운트되게 하려고
    4. D. epoch 수를 줄이려고
    정답 B. scope() 내부에서 만든 변수가 분산 변수로 생성되어 각 replica에 미러링·동기화된다. scope 밖에서 만들면 분산이 올바르게 적용되지 않는다. loss·버킷·epoch와는 무관하다.
  3. replica를 4개로 늘려 동기 data parallelism을 할 때 일반적으로 함께 조정하는 것은?
    1. A. loss function 종류
    2. B. global batch size를 키우고 그에 맞춰 learning rate를 조정
    3. C. 모델의 layer 개수 자동 증가
    4. D. optimizer를 SGD로 강제 변경
    정답 B. data parallelism에서 global batch size는 replica들에 분배되므로 replica가 늘면 global batch size를 키우고 보통 learning rate도 그에 맞춰 조정한다. loss·layer·optimizer를 강제로 바꿀 필요는 없다.
  4. 동기 분산 학습에서 각 replica의 gradient를 모아 합산·평균하여 모든 replica가 동일한 가중치 업데이트를 받도록 하는 연산은?
    1. A. all-reduce
    2. B. softmax
    3. C. dropout
    4. D. early stopping
    정답 A. all-reduce는 분산된 gradient를 집계·동기화해 모든 replica가 같은 업데이트를 적용하게 하는 핵심 통신 연산이다. softmax·dropout·early stopping은 분산 동기화와 관계없는 학습 구성요소다.
  5. MultiWorkerMirroredStrategy로 여러 머신에서 학습할 때 클러스터 구성(워커 목록·역할)을 각 워커에 알려주는 것은?
    1. A. AIP_MODEL_DIR
    2. B. TF_CONFIG 환경변수
    3. C. strategy.scope() 인자
    4. D. compile()의 metrics
    정답 B. MultiWorkerMirroredStrategy는 TF_CONFIG 환경변수로 워커 주소와 역할(task index 등)을 인식해 통신을 구성한다. AIP_MODEL_DIR은 모델 저장 경로, scope·metrics는 클러스터 구성과 무관하다.

복습: Running Pipelines on Agent Platform 2.5

  • 15m 📝 component·DAG·artifact·lineage 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답 복습 및 compile/submit 재현
📌 핵심 포인트
  • KFP(Kubeflow Pipelines) 기반 파이프라인은 ML 워크플로를 component 단위로 모듈화하고 이를 DAG로 연결해 단계 의존성을 정의한다.
  • 각 component는 독립 실행되며 입력·출력으로 parameter(스칼라 값)와 artifact(데이터셋·모델 등 큰 객체)를 주고받는다.
  • lineage(계보)는 어떤 데이터·코드·파라미터가 어떤 산출물을 만들었는지 추적해 재현성과 거버넌스를 제공한다.

복습: Feature Store: Streaming Ingestion SDK

  • 15m 📝 핵심 개념 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답·streaming 시나리오 재현
📌 핵심 포인트
  • Feature Store는 online serving(저지연·실시간 예측용 최신 feature 조회)과 offline serving(대량 batch·training용 historical feature 조회)을 분리 제공하며, 두 경로가 동일한 feature 정의를 공유해 training-serving skew를 줄인다.
  • Streaming ingestion(write/import streaming SDK)은 이벤트 발생 즉시 단건/소량 feature 값을 online store에 기록해 feature freshness(최신성)를 높이며, 주기적 batch ingestion만으로는 따라가기 힘든 실시간 시나리오(예: 직전 클릭, 잔액)에 적합하다.
  • Training-serving skew는 학습 시 사용한 feature 값/계산 로직과 serving 시점의 값/로직이 달라 발생하며, Feature Store로 feature 계산을 중앙화·재사용하면 동일 로직이 양쪽에 적용되어 skew를 완화한다.
  • Feature reuse/sharing은 한 번 정의·검증한 feature를 여러 팀·여러 모델이 공유해 중복 engineering을 줄이고 일관성을 높이는 것으로, entity type과 feature 메타데이터로 발견·재사용을 돕는다.
  • Point-in-time correctness가 중요한 offline 조회에서는 label 시점 이전의 feature 값만 join해 data leakage(미래 정보 누수)를 방지해야 한다.
  • Feature freshness와 ingestion 비용·복잡도는 trade-off 관계로, 실시간성이 꼭 필요한 feature에만 streaming을 쓰고 나머지는 batch로 처리하는 설계가 일반적이다.

복습: Get Started with Agent Studio

  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답·agent 구성 재현
📌 핵심 포인트
  • Agent Studio는 low-code 환경에서 agent를 빠르게 prototype하고 평가·배포하도록 돕는 도구로, 코드를 많이 작성하지 않고도 reasoning·instruction·tool을 조합한 agent를 구성할 수 있다.
  • agent는 본질적으로 LLM의 reasoning(추론) + instructions(역할·행동 지침/시스템 프롬프트) + tools(외부 함수·API·검색 등 호출 능력)의 결합으로, 단순 단발 프롬프트 응답과 달리 도구를 사용해 작업을 수행한다.
  • instructions는 agent의 목적·말투·금지사항·작업 절차를 정의해 일관된 행동을 유도하며, 모호한 지침은 예측 불가한 결과를 낳으므로 명확·구체적으로 작성한다.
  • tools는 모델이 자체 지식만으로 할 수 없는 작업(최신 데이터 조회, 계산, 외부 시스템 액션)을 수행하게 하며, agent는 어떤 tool을 언제 호출할지 reasoning으로 결정한다.
  • 개발 흐름은 prototype(빠른 구성·실험) → evaluate(품질·정확성·안전성 평가) → deploy(운영 배포)의 반복 cycle로 진행되며, 배포 전 평가로 회귀와 위험을 점검한다.
  • Agent Platform(구 Vertex AI) 위에서 동작해 모델·평가·배포 인프라와 통합되며, 평가 단계에서 정의된 기준으로 응답 품질을 비교·개선한다.

복습: Build an LLM and RAG-based Chat Application with AlloyDB and Agent Platform

  • 15m 📝 핵심 개념(RAG 파이프라인) 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답·retrieve→augment→generate 재현
📌 핵심 포인트
  • RAG(Retrieval-Augmented Generation)는 retrieve(관련 문서 검색) → augment(검색 내용을 프롬프트에 결합) → generate(근거 기반 응답 생성)의 3단계로, 모델 외부의 최신·도메인 지식을 답변에 주입해 hallucination을 줄인다.
  • 문서는 chunking으로 적절한 크기 조각으로 나눠 embedding하며, chunk가 너무 크면 무관 내용이 섞이고 너무 작으면 문맥이 끊기므로 크기·overlap을 조절한다.
  • embedding은 텍스트를 의미를 담은 vector로 변환하며, 질의도 같은 embedding 공간으로 변환해 vector similarity(예: cosine/내적)로 의미적으로 가까운 chunk를 검색한다.
  • AlloyDB는 vector(예: pgvector 기반) 저장·유사도 검색을 지원하는 데이터베이스로, embedding을 저장하고 ANN/유사도 쿼리로 관련 chunk를 retrieve하는 vector store 역할을 한다.
  • grounding은 검색된 실제 근거에 응답을 묶어 사실성을 높이는 것으로, 근거가 없으면 '모른다'고 답하게 하면 hallucination을 추가로 줄일 수 있다.
  • 검색 품질(관련 chunk를 잘 가져오는지)이 최종 답변 품질을 좌우하므로, embedding 모델·chunking 전략·top-k 검색 개수가 핵심 튜닝 지점이다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. RAG의 표준 처리 순서로 옳은 것은?
    1. A. generate → retrieve → augment
    2. B. retrieve → augment → generate
    3. C. augment → generate → retrieve
    4. D. retrieve → generate → augment
    정답 B. RAG는 관련 문서를 먼저 retrieve하고, 그 내용을 프롬프트에 augment한 뒤 근거 기반으로 generate한다. 검색 없이 먼저 생성하면 근거 주입 효과가 사라진다.
  2. RAG가 LLM의 hallucination을 줄이는 주된 메커니즘은?
    1. A. 모델 파라미터를 늘린다
    2. B. 검색된 실제 근거에 응답을 grounding한다
    3. C. temperature를 1.0으로 높인다
    4. D. 출력 토큰 수를 제한한다

복습: Mitigate Bias with MinDiff in TensorFlow

  • 15m 📝 핵심 개념(fairness 정의·MinDiff) 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답·trade-off 재현
📌 핵심 포인트
  • fairness 작업은 먼저 measure(슬라이스별 지표로 편향 측정)하고 그다음 mitigate(완화)하는 순서로 진행하며, 측정 없이 완화하면 효과·부작용을 검증할 수 없다.
  • MinDiff는 in-processing 기법으로, 기존 손실에 추가 손실(regularization)을 더해 두 slice(예: 민감 그룹) 간 특정 예측 분포의 차이를 줄이도록 학습한다.
  • demographic parity는 그룹과 무관하게 positive 예측 비율이 같기를 요구하고, equal opportunity는 실제 positive(자격이 있는) 대상에 대한 true positive rate가 그룹 간 같기를 요구하는 등 fairness 정의에 따라 목표가 다르다.
  • slice(하위 그룹) 단위 평가가 핵심으로, 전체 정확도는 좋아도 특정 그룹에서 성능·오류율이 크게 나쁠 수 있으므로 그룹별 지표를 따로 본다.
  • fairness 완화는 보통 accuracy-fairness trade-off를 동반해 전체 정확도가 다소 하락할 수 있으며, MinDiff loss의 가중치로 이 균형을 조절한다.
  • MinDiff는 비교하려는 두 그룹의 예시 묶음을 추가 입력으로 받아 그들의 점수 분포 차이를 penalize하므로, 어떤 slice를 정렬할지 정의가 중요하다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. 공정성 작업에서 일반적으로 권장되는 순서는?
    1. A. 먼저 mitigate하고 그다음 measure
    2. B. 먼저 measure하고 그다음 mitigate
    3. C. measure 없이 바로 deploy
    4. D. accuracy만 보고 fairness는 보지 않음
    정답 B. 편향을 슬라이스별로 측정해야 무엇을 얼마나 완화할지, 완화 후 개선/부작용을 확인할 수 있다. 측정 없는 완화나 정확도만 보는 접근은 공정성 검증을 불가능하게 한다.
  2. MinDiff를 분류상 가장 정확히 설명한 것은?
    1. A. 데이터를 재수집하는 pre-processing 기법
    2. B. 학습 후 임계값만 조정하는 post-processing 기법
    3. C. 학습 중 추가 손실을 더해 그룹 간 예측 분포 차이를 줄이는 in-processing 기법
    4. D. 모델 구조를 자동 탐색하는 NAS 기법
    정답 C. MinDiff는 학습 단계에서 기존 손실에 추가 손실을 더하는 in-processing 방식이다. 데이터 재수집(pre-)이나 임계값 조정(post-)과 다르며, 구조 탐색(NAS)과도 무관하다.
  3. '자격을 갖춘(실제 positive) 지원자 중 승인 비율(TPR)이 그룹 간 같아야 한다'는 fairness 정의는?
    1. A. demographic parity
    2. B. equal opportunity
    3. C. individual fairness
    4. D. calibration
    정답 B. 실제 positive에 대한 TPR의 그룹 간 동등성은 equal opportunity의 정의다. demographic parity는 실제 자격과 무관하게 positive 예측 비율의 동등성을 요구하므로 다르다.
  4. 전체 test 정확도는 92%로 높은데도 fairness 점검이 필요한 이유로 가장 적절한 것은?
    1. A. 전체 정확도가 높으면 모든 그룹이 공정하다
    2. B. 특정 slice(하위 그룹)에서 오류율이 크게 나쁠 수 있어 그룹별 지표를 봐야 한다
    3. C. 정확도가 높으면 데이터가 충분하다는 뜻이다
    4. D. fairness는 정확도와 항상 동일하다
    정답 B. 집계된 전체 정확도는 소수 그룹의 큰 성능 격차를 가릴 수 있어 slice별 평가가 필요하다. 높은 전체 정확도가 그룹 공정성이나 데이터 충분성을 보장하지는 않는다.
  5. MinDiff loss의 가중치를 크게 키웠을 때 일반적으로 기대되는 효과는?
    1. A. 그룹 간 격차는 줄지만 전체 정확도가 다소 하락할 수 있다
    2. B. 정확도와 공정성이 항상 동시에 무한히 좋아진다
    3. C. 공정성과 정확도 모두 변화가 없다
    4. D. 모델이 학습 자체를 멈춘다
    정답 A. MinDiff 가중치를 키우면 분포 차이 완화가 강해져 격차는 줄지만 accuracy-fairness trade-off로 정확도가 떨어질 수 있다. 두 지표가 무한히 동시에 좋아지지는 않으며 학습이 멈추지도 않는다.

복습: Explaining an Image Classification Model with Agent Platform Explainable AI

  • 15m 📝 핵심 개념(attribution 기법별 차이) 정리
  • 25m ❓ 퀴즈 5문항 풀이
  • 10m 🔁 오답·IG/XRAI 비교 재현
📌 핵심 포인트
  • feature attribution은 모델의 특정 예측에 각 입력 feature(이미지의 경우 픽셀·영역)가 얼마나 기여했는지 점수로 설명해, 디버깅과 신뢰(trust) 확보에 쓰인다.
  • Integrated Gradients(IG)는 baseline(예: 검은 이미지)에서 실제 입력까지 경로를 따라 gradient를 적분(누적)해 각 픽셀 기여도를 구하는 gradient 기반 기법으로, baseline 선택이 결과에 영향을 준다.
  • XRAI는 IG 기반 신호를 픽셀 단위가 아닌 영역(region) 단위로 묶어 '이미지의 어느 영역이 중요한지'를 보여주는 region-based 기법으로, 이미지 해석에 직관적이다.
  • sampled Shapley는 feature를 빼고/넣어가며 기여도를 추정하는 perturbation 기반 기법으로 gradient가 필요 없지만 계산 비용이 크고, gradient 기반(IG/XRAI)은 미분 가능한 모델에서 효율적이다.
  • attribution은 모델이 '엉뚱한 영역'(예: 배경·워터마크)을 보고 맞히는 spurious correlation을 드러내 디버깅을 돕는다.
  • attribution은 예측에 대한 기여 설명일 뿐 인과관계나 정답 보장이 아니며, baseline·기법 선택에 따라 해석이 달라질 수 있음을 유의해야 한다.
❓ 퀴즈 5문항 (질문 클릭 → 보기 → 정답)
  1. Integrated Gradients가 픽셀 기여도를 계산하는 방식으로 옳은 것은?
    1. A. 단일 입력에서의 gradient 한 번만 사용한다
    2. B. baseline에서 입력까지 경로를 따라 gradient를 적분(누적)한다
    3. C. feature를 무작위로 제거하며 점수 변화를 측정한다
    4. D. 모델 가중치를 직접 시각화한다
    정답 B. IG는 baseline에서 실제 입력으로 가는 경로상의 gradient를 적분해 기여도를 구한다. 무작위 제거는 Shapley 계열 perturbation 방식이고, 단일 gradient나 가중치 시각화와는 다르다.
  2. 이미지 분류 설명에서 픽셀 단위가 아니라 '의미 있는 영역' 단위로 중요도를 보여주는 기법은?
    1. A. XRAI
    2. B. sampled Shapley
    3. C. L2 regularization
    4. D. batch normalization
    정답 A. XRAI는 attribution 신호를 region 단위로 묶어 영역 기반 설명을 제공해 이미지 해석에 직관적이다. sampled Shapley는 feature 단위 perturbation 기법이고, 나머지는 설명 기법이 아닌 학습 기법이다.
  3. gradient를 사용할 수 없는(미분 정보 활용이 어려운) 상황에서 attribution을 구하려 할 때 더 적합한 기법은?
    1. A. Integrated Gradients
    2. B. XRAI
    3. C. sampled Shapley
    4. D. saliency gradient
    정답 C. sampled Shapley는 feature를 넣고/빼며 기여도를 추정하는 perturbation 기반이라 gradient가 필요 없다. IG·XRAI·saliency는 모두 gradient에 의존하므로 이 상황에 부적합하다.

    1. A. 정확도를 자동으로 올려준다
    2. B. spurious correlation 같은 잘못된 근거를 드러내 디버깅·신뢰 점검을 돕는다
    3. C. 학습 속도를 빠르게 한다
    4. D. 모델 파라미터를 줄여준다
    정답 B. attribution은 모델이 엉뚱한 영역을 근거로 삼는지 드러내 디버깅과 신뢰 확보에 쓰인다. 정확도 향상·학습 가속·파라미터 축소는 attribution의 직접 기능이 아니다.

    2. B. baseline은 '정보 없음' 기준점으로, 선택에 따라 attribution 해석이 달라질 수 있다
    3. C. baseline은 항상 흰색 이미지여야만 한다
    4. D. baseline은 모델 가중치를 의미한다
    정답 B. IG의 baseline은 기여도를 측정하는 기준점이라 무엇을 쓰느냐에 따라 결과 해석이 달라진다. 영향이 없다거나 반드시 흰색이어야 한다는 것은 틀리며, baseline은 가중치가 아니라 입력 기준값이다.

📌 핵심 포인트
  • Gemini API는 유해 콘텐츠 카테고리별 safety filter를 제공해, 입력·출력에서 위험 콘텐츠를 탐지하고 설정한 threshold(차단 민감도)에 따라 차단·허용을 조절한다.
  • category threshold를 엄격하게 하면 안전성은 높아지지만 정상 요청까지 막혀 utility가 떨어지는 safety-utility trade-off가 발생하므로, 용도에 맞게 균형을 잡는다.
  • prompt injection은 사용자/외부 입력이 시스템 지침을 무시·탈취하도록 유도하는 공격으로, 입력 검증·지침 분리·출력 점검 등 다층 방어가 필요하다.
  • Model Armor 같은 보호 계층은 모델 앞단/주변에서 프롬프트·응답을 검사해 prompt injection·유해 콘텐츠·민감정보 노출 등을 추가로 걸러내는 방어 장치 역할을 한다.
  • responsible AI 관점에서 안전은 단일 필터가 아니라 측정·정책·필터·모니터링이 결합된 다층 접근으로 다루며, 안전 설정은 지속적으로 평가·조정한다.
  • safety filter는 확률적이라 완벽하지 않으므로 false positive(과차단)·false negative(누락)를 모두 고려해, 위험도가 높은 도메인일수록 보수적으로 설정한다.

  2. prompt injection 공격을 가장 잘 설명한 것은?
    1. A. 모델 가중치를 직접 수정하는 공격

  3. category별 safety filter 설정에 대한 올바른 이해는?
    1. A. 모든 카테고리에 항상 동일한 단일 threshold만 써야 한다
    2. B. 유해 카테고리별로 threshold를 두어 차단 민감도를 조절할 수 있다
    3. C. filter는 입력에는 적용되지 않고 출력에만 적용된다
    4. D. threshold는 안전성과 무관한 성능 설정이다
    정답 B. safety filter는 유해 카테고리별 threshold로 차단 민감도를 조절하도록 설계된다. 단일 고정 threshold 강제나 출력 전용이라는 설명은 부정확하며, threshold는 명백히 안전성 설정이다.
  4. 외부 문서를 그대로 프롬프트에 넣는 RAG형 agent에서 prompt injection 위험을 줄이는 방어로 가장 적절하지 않은 것은?
    1. A. 입력/검색 콘텐츠 검증과 시스템 지침의 명확한 분리
    2. B. 출력 점검 및 safety filter 적용
    3. C. Model Armor 같은 보호 계층 추가
    4. D. 안전 설정을 모두 끄고 모델 응답을 그대로 신뢰
    정답 D. injection 방어는 입력 검증·지침 분리·출력 점검·보호 계층 등 다층 접근이 핵심이다. 안전 설정을 끄고 그대로 신뢰하는 것은 오히려 공격에 그대로 노출되므로 방어가 아니다.

    1. A. utility를 위해 모든 filter를 비활성화한다
    2. B. 위험도를 고려해 보수적으로(더 엄격하게) 설정하고 지속 모니터링한다
    3. C. 한 번 설정하면 다시 평가하지 않는다
    4. D. safety는 정확도와 무관하므로 무시한다
    정답 B. 고위험 도메인은 과차단을 감수하더라도 보수적 설정과 지속 모니터링으로 위험을 낮추는 것이 responsible AI에 부합한다. filter 비활성화나 일회성 설정·무시는 위험을 키운다.
